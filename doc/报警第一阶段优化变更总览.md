# 报警第一阶段优化变更总览

## 文档定位

本文是 `hpis-alarm` 第一阶段报警分片与消警可靠化优化的中文总入口。后续阅读优先看本目录下中文命名文档，旧英文命名文档保留为历史草案和配套记录，不再作为最新口径的唯一来源。

## 第一阶段完成内容

本阶段围绕“报警可分片、消警不丢、压测可定位、空闲少打扰、批量链路可回滚”完成了七类改造：

1. 时间 + 容量分片：业务表从固定 `alarm_id % 5` 转向 `alarm_yyyyMM_nn` 形式，按报警开始时间定位月份，按容量阈值切月内子表。
2. `alarm_id` 路由化：内部 `alarm_id` 保持 `bigint`，编码中包含日期、slice、worker、rowNo 等信息，用于按 ID 直达分片。
3. cid 热点路由：`alarm_cid_index` 只服务 active/hot 生命周期；长期历史不再通过 cid 快速定位，避免全量路由表无限增长。
4. stop 可靠化：MQ stop 先落 `alarm_stop_event`，后台 worker 批量关闭业务表、cid route 和 event 状态，side effect 独立执行。
5. 运行降噪：stop worker 空闲后暂停数据库轮询，收到新 stop 后唤醒；SQL 日志改为 `alarm.sql-log.*` 分层开关。
6. insert/stop 批量化：新增 `alarm.batch.*` 开关，stop route 查询从循环单查改为 `IN` 批量查询，insert 新增内部 `insertAlarms` 批量 API；外部 MQ/Controller 单条入口和 push 消息体保持不变。
7. MQ start consumer batch：新增 `RabbitMQAlarmBatchListener`，用 Spring AMQP 原生 consumer batch 收集一批消息后批量持久化 start；同批 start 先落库，再处理 stop/status，ack/nack 仍逐条执行。

## 当前边界

- 当前不删除业务报警历史，只清理 hot/stale cid 路由和已完成 stop event。
- cid 只服务 active/hot 生命周期；关闭后过一段时间会被物理清理。
- 长期历史查询应走内部 `alarm_id` 或时间范围。
- stop 允许秒级最终一致，但最终必须写入 `alarm_endTime`。
- side effect 可以延迟，但不能阻塞核心消警。
- stop worker 默认连续 3 轮空处理后进入 idle pause，靠新 stop 入库唤醒，并保留 60 秒兜底扫描。
- stop 批量链路中 true route 缺失会转 `FAILED/ROUTE_MISSING`；关闭 `alarm.batch.stopEnabled` 可回到旧单条链路和旧 pending 补偿语义。
- `insertAlarm(JSONObject)` 外部入口不变；`insertAlarms(List<JSONObject>)` 是内部扩展 API，批量失败会回滚批量事务后逐条兜底。
- MQ start consumer batch 默认关闭；开启 `alarm.batch.insertConsumerBatchEnabled=true` 后旧 `RabbitMQAlarmListener` 条件关闭，`RabbitMQAlarmBatchListener` 接管同一个 `alarm_queue`。
- consumer batch 第一阶段只批量 start 持久化；同批 start 会先处理，同批 stop/status 随后按旧单条链路处理并逐条 ack/nack。
- consumer batch 不移动 `breakLine(...)` / `breakLineUnitAction(...)` 到 afterCommit；批量失败拆单时复用 context，避免重新组装导致额外重复触发，但首次批量尝试前的远程副作用仍可能早于 DB 提交。
- 旧单条 listener 和 consumer batch 都不保证同一 `alarmCid` 严格顺序，沿用当前 Rabbit 并发消费下的最终一致语义。
- SQL info 默认不打印；生产临时排查时通过 `alarm.sql-log.enabled=true` 和 `mode` 控制范围。
- 多实例部署必须为每个实例配置不同 `alarm.sharding.id.workerId`。
- 后续每轮代码修改必须遵守根目录 `agent.md`：关键业务代码补充中文风险注释，并新增专门修改文档；本约束只新增协作规范，不改变运行期行为。

## MQ start consumer batch 风险点与迭代计划

> 2026-05-21 1000 条 start-only 压测显示，历史 `AlarmInsertAggregateService.submitAndWait` 方案功能可用但吞吐不达预期：listener 线程同步等待导致实际 batchSize 被当前消费者数限制，配置 200 时实际多为 10。2026-05-23 已清理该自研聚合器，后续只保留旧单条 listener 与 Spring AMQP consumer batch 两条入口。
>
> 2026-05-22 1000 条 start-only 压测显示，Spring AMQP consumer batch 方案功能通过：`insertConsumerBatchEnabled=true`、`batchSize=100`、`receiveTimeoutMs=50`、`concurrency=2-4`、`prefetch=100` 时，实际 10 批、每批 100 条，`ack=1000`、`nack=0`，PERSIST 首尾耗时 2723ms，估算 367.24/s。对比旧单条 listener 104.22/s、旧 submitAndWait 聚合 38.76/s，第一阶段生产灰度应优先走 consumer batch。

### 风险点

1. Redis 去重复用：断线告警在 prepare 阶段执行 `tryAcquire`；批量事务失败后必须复用 `AlarmInsertContext` 拆单，不能重新 prepare，否则会错误跳过或重复触发副作用。
2. 批量事务回滚：跨多个 suffix 的 batch flush 是一个事务，任一 suffix 写失败会整批回滚；`fallbackSingleOnBatchError=false` 时整批返回 FAIL 并 requeue。
3. 远程副作用允许重复触发：第一阶段不移动 `breakLine`、`breakLineUnitAction` 到 afterCommit，批量失败和 MQ 重投场景仍可能重复触发远程状态同步，业务已接受该边界。
4. Rabbit 并发顺序：旧 listener `concurrency="10-36"` 或 batch listener `insertConsumerBatchConcurrency` 下，同一 `alarmCid` 的 start 不保证严格顺序，不新增同 cid 串行化。
5. push 兼容性：batch 成功后仍逐条 afterCommit 发布原 JSON 结构；如果未来改 payload，必须另开兼容性评审。
6. consumer batch 混合消息：同一个 batch 可能同时包含 start、stop、status；当前实现先处理 start，再处理 direct 消息，降低 stop 先查不到 route 的误判窗口。
7. consumer batch ack 粒度：只要存在部分成功/失败，就必须逐条 ack/nack，不能使用 `multiple=true` 批量确认。
8. consumer batch 配置误差：`prefetch < batchSize` 或 `receiveTimeoutMs` 过小会导致实际 batchSize 偏小，必须用日志观测 `actualBatchSize`。

### 迭代计划

1. 第一阶段：`insertConsumerBatchEnabled=false` 默认关闭，灰度时手动改 true；`batchSize=100`、`receiveTimeoutMs=50`、`concurrency=2-4`、`prefetch=100` 起步，已完成 1000 条 start-only 验证。
2. 第二阶段：如果 consumer batch 吞吐达标，再接 MQ backlog 或消费 lag 指标，决定何时开启 batch listener 或调整 batch 参数。
3. 第三阶段：批量设备解析、报警配置解析和 Redis 读优化，减少 prepare 阶段逐条远程/缓存访问。
4. 第四阶段：压测后再评估多分区 consumer batch 或 backlog 自适应参数；不再恢复自研内存 submitAndWait 聚合器。

## 后续优化方向

压测显示本阶段已把逐条 `closeRoute`、`markApplied`、`markDone` 降为几十批，剩余主要瓶颈不再是这些逐条更新，而是：

- RabbitMQ listener 消费节奏和并发配置。
- start 插入链路仍存在多表写入成本，电解槽场景额外写扩展表更明显。
- 内部 `insertAlarms` 已支持按 suffix 批量持久化；MQ 高峰期优先通过 Spring AMQP consumer batch 调用批量持久化。
- stop event 每条 stop 消息仍需要一次可靠 upsert。
- 电解槽 `EC_ECTYPE_DELETE` 副作用量大，后续可考虑按 alarmId 批量执行。
- 空闲暂停只降低无任务时的数据库轮询，不能解决高峰期 MQ 消费和 stop upsert 吞吐问题。
