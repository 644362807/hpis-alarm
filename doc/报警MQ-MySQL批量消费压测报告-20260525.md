# 报警 MQ-MySQL 批量消费压测报告 - 20260525

## 结论

本轮 `MIXED + ALTERNATE + 100000` 分表上限门禁通过。最终有效门禁使用干净测试月份 `202608`，`alarm.sharding.maxRowsPerSlice=10000`，生成 `alarm_202608_00` 到 `alarm_202608_09` 共 10 个容量分片，每片本轮真实主表行数均为 10000。

当前代码可以定位为“生产化候选 + 需要灰度验证”。本轮未发现主表/电解槽扩展表 `alarm_id` 不一致、MQ 持续重投、MySQL 半成品数据或分片跨表 suffix 不一致问题。灰度阶段仍需重点观察 stop event 重试、`ROUTE_MISSING` 恢复率、分片元数据状态、DB 写放大和 10w 以上长时间运行吞吐。

## 测试说明

第一次按计划使用 `202607` 执行时，MQ 到 DB 闭环通过，但分片上限门禁未通过。原因是本地 Nacos `hpis-alarm-dev.yml` 中已有 `alarm.sharding.maxRowsPerSlice: 5000000`，且 bootstrap 配置优先级高于 `commandLineArgs`，导致命令行覆盖的 `10000` 未生效。`202607` 因此只写入 `alarm_202607_00`，不能作为分表上限样本。

修正方式：临时将 Nacos 中 `alarm.sharding.maxRowsPerSlice` 调整为 `10000`，重启 hpis-alarm 后确认运行态属性来源为 `bootstrapProperties-hpis-alarm-dev.yml` 且值为 `10000`。为避免删除历史业务表和污染样本，最终门禁改用未使用的独立月份 `202608`。

## 风险与生产化建议

1. Nacos 配置优先级必须纳入发布说明。分片上限、批消费开关、stub 开关不能只靠命令行覆盖判断，发布前应通过 `/actuator/env` 或等价方式确认运行态值。
2. `ROUTE_MISSING` 可重试路径本轮生效，但灰度仍要监控 `alarm_stop_event.PENDING/FAILED/retry_count/last_error`，特别是 start/stop 交替高峰。
3. 分片元数据建议增强：满片后将 `status` 更新为 `FULL`，并增加巡检 SQL 对比 `current_rows/max_rows` 和物理表真实行数。
4. 当前 10w 分片闭环吞吐约 `153 rows/s`，瓶颈仍在 stop worker 与 MySQL 写放大。生产灰度先保持 `consumer concurrency=2-4`，优先观察 DB、stop worker 和 route 查询耗时，再决定是否调大并发。
5. 缺参毒消息路径不属于本轮 10w 容量样本输入；代码和单测已覆盖 `DROP + ack`，灰度仍需监控 nack/requeue、半成品主表和电解槽扩展表空 ID。

## 最终判定

本轮 `MIXED + ALTERNATE + 100000` 分表上限门禁通过。当前完成代码可以作为生产化候选进入灰度，但不建议直接全量发布；灰度前必须锁定 Nacos 配置、保留 stop event 与分片巡检 SQL、并对 10w 以上长时间流量继续观察。
