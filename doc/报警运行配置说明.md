# 报警运行配置说明

## 文档定位

本文说明 `hpis-alarm` 当前运行期 `alarm.*` 配置的真实绑定情况、默认值、是否必须配置、执行入口和上线风险。

结论先说清楚：

- `alarm.sharding.*` 是有效配置，绑定到 `AlarmShardProperties`，控制时间 + 容量分片、cid 热点路由和内部 alarm_id 编码。
- `alarm.stop-worker.*` 是有效配置，绑定到 `AlarmStopWorkerProperties`，控制可靠消警 worker、side effect worker 和 APPLIED 清理。
- `alarm.sql-log.*` 是有效配置，绑定到 `AlarmSqlLogProperties`，控制自定义 MyBatis SQL 日志拦截器是否打印 SQL。
- `alarm.profile.*` 当前代码没有任何读取入口，属于无效配置或历史压测残留；保留在 Nacos 不会生效。
- 本地 `bootstrap.yml` 只负责连接 Nacos。实际运行配置通常来自 Nacos 的 `hpis-alarm-${profile}.yml` 或共享 `application-${profile}.yml`。

## 推荐注释版配置

```yaml
alarm:
  sharding:
    # 是否启用 hpis-alarm Java API 时间 + 容量分片。
    # false 或缺失时，不创建新分片 DataSource，也不会启用 cid lifecycle job、stop worker。
    # 生产开启前必须确认旧 alarm_id % 5 inline 分片规则已关闭，避免两套分片规则同时生效。
    enabled: true

    # 单个月内单个子表最大预占行数。
    # 到达阈值后，当前月份会切到下一个 nn 子表，例如 alarm_202605_01。
    # 代码有边界校验：必须 > 0，且不能超过 8388608。
    maxRowsPerSlice: 5000000

    # 预创建未来月份数量。
    # 代码实际会创建当前月 + 未来 N 个月的 00 子表。
    preCreateMonths: 2

    # 本机每次从 alarm_shard_slice 预占的 rowNo 号段大小。
    # 值越大，数据库锁竞争越低，但服务异常退出时可能留下更多未使用空洞。
    # 配置小于 1 时，代码会用 Math.max(1, value) 兜底为 1。
    allocationSegmentSize: 1000

    # 迁移期是否把旧 alarm_0..4 / alarm_handle_0..4 / alarm_electrolytic_cell_0..4 纳入查询兜底。
    # 旧数据未迁移完成时保持 true；迁移完成并回填路由后建议改 false，减少查询扫描范围。
    includeLegacyTables: true

    id:
      # 内部 alarm_id 编码中的 workerId，占 8 bit，有效范围 0..255。
      # 单实例可用 0；多实例生产部署必须为每个实例配置不同值，否则极端情况下可能出现 ID 冲突。
      workerId: 0

    cidIndex:
      # 是否打印 cid 路由生命周期定时任务的普通完成日志。
      # 生产默认 false，只保留异常日志；排查 hot/stale 流转时可临时开启。
      logEnabled: false

      # ACTIVE hot 路由保留小时数。超过后，低流量任务转入 alarm_cid_stale_index。
      hotHours: 24

      # stale 路由保留天数。超过后，任务会先关闭业务报警，再关闭路由。
      staleExpireDays: 30

      # CLOSED hot/stale 路由清理批大小。
      cleanupBatchSize: 1000

      # hot 转 stale 每批处理数量。
      transferBatchSize: 1000

      # CLOSED 路由物理清理任务 cron。
      cleanupCron: "0 0 2 * * ?"

      # hot 转 stale 任务 cron。
      transferCron: "0 10 2 * * ?"

      # stale 超时关闭任务 cron。
      staleTimeoutCron: "0 20 2 * * ?"

  stop-worker:
    # 是否打印 stop worker 普通完成日志。
    # 生产默认 false，避免高频调度刷屏；排查处理节奏时可临时开启。
    logEnabled: false

    # PENDING stop event 数量 <= 该值时视为低流量，允许执行 APPLIED 清理。
    lowWatermark: 500

    # PENDING stop event 数量 >= 该值时进入高流量模式，核心消警 worker 会加大批量和循环次数。
    highWatermark: 5000

    # 正常流量每批处理 PENDING stop event 数量。
    normalBatchSize: 500

    # 高流量每批处理 PENDING stop event 数量。
    highBatchSize: 2000

    # 核心 stop worker 和 side effect worker 调度间隔，单位毫秒。
    normalIntervalMs: 1000

    # 当前代码没有直接作为 @Scheduled 间隔使用。
    # 高流量实际通过 highBatchSize + maxParallelism 提升单轮处理能力。
    highIntervalMs: 100

    # 高流量模式下单轮最多连续处理批次数。
    maxParallelism: 4

    # 是否生成并执行消警后的副作用事件，例如设备在线状态恢复、电解槽扩展清理。
    # 关闭后只保证核心 alarm_endTime 和 cid route 关闭。
    sideEffectEnabled: true

    # APPLIED stop event 保留分钟数，到期后在低流量窗口清理。
    appliedRetentionMinutes: 30

    # APPLIED stop event 清理批大小。
    cleanupBatchSize: 1000

    # APPLIED stop event 清理扫描间隔，单位毫秒。
    cleanupIntervalMs: 60000

    # FAILED 事件保留天数。当前代码只记录该配置，不会自动删除 FAILED。
    failedRetentionDays: 7

    # true 表示只在低流量模式清理 APPLIED stop event。
    cleanupOnlyLowTraffic: true

    # 单条 stop event 或 side effect event 最大重试次数，达到后转 FAILED。
    maxRetry: 5

    # 是否启用 stop worker 空闲暂停。
    # true 时，连续空轮次后 @Scheduled 仍触发，但只做内存判断，不再每秒查 stop event 表。
    idlePauseEnabled: true

    # 连续多少轮核心消警 + 副作用都无处理量后进入 idle pause。
    idleConfirmCount: 3

    # idle pause 后低频兜底查库间隔，单位毫秒。
    # 用于防止重启、漏唤醒、多实例场景下 PENDING 长期卡住。
    idleProbeIntervalMs: 60000

  sql-log:
    # 自定义 MyBatis SQL info 日志总开关。
    # false 时不打印任何 SQL info；warn/error 异常日志仍保留。
    enabled: false

    # SQL 打印模式：
    # all：打印所有 SQL，等同旧行为，生产不建议长期开启。
    # alarm-write：只打印报警主链路写 SQL，并额外打印慢 SQL；stop-worker SQL 默认排除。
    # slow：只打印慢 SQL，stop-worker SQL 默认排除。
    mode: alarm-write

    # 是否打印参数和 Full SQL。
    # 生产涉及敏感字段时建议 false。
    printParam: false

    # enabled=true 时，是否允许慢 SQL 日志。
    slowEnabled: true

    # 慢 SQL 阈值，单位毫秒。
    slowMs: 200

  profile:
    # 当前代码没有读取 alarm.profile.* 的入口。
    # 这组配置不会影响运行行为，建议从生产 Nacos 删除，或后续恢复 profiling 组件后再启用。
    enabled: false
    methodEnabled: false
    runId: batch-electro-20k-20260514-001
    logIntervalMs: 5000
    slowSqlMs: 50
    summaryTopN: 30
    methodSlowMs: 100
    methodSlowLogEnabled: true
```

## 执行入口

### 配置加载入口

- `bootstrap.yml`：只配置服务名、profile、Nacos 地址和共享配置。
- Nacos：实际放置 `alarm.*` 的地方，通常是 `hpis-alarm-${profile}.yml` 或共享的 `application-${profile}.yml`。
- `HpisAlarmApplication`：通过 `@EnableScheduling` 开启定时任务。

### 分片入口

- `AlarmDynamicShardingConfig`
  - 条件：`alarm.sharding.enabled=true`
  - 作用：创建 Java API ShardingSphere 数据源、`AlarmMonthlySliceTableManager`、`AlarmCidIndexService`、`AlarmIdCodec`、`AlarmTimeCapacityShardingAlgorithm`。

- `AlarmServiceImpl.insertAlarm`
  - 新增报警入口。
  - 调用 `prepareAlarmShard` 分配 `tableSuffix`，通过 `AlarmShardContext` 固定本次 `alarm/alarm_handle/alarm_electrolytic_cell` 三张表落到同一分片。
  - 写入主表后调用 `AlarmCidIndexService.saveActiveHotRoute` 保存 hot cid 路由。

- `AlarmTimeCapacityShardingAlgorithm`
  - 查询/更新路由入口。
  - 支持按 `AlarmShardContext`、内部 `alarm_id`、外部 `alarm_cid`、时间范围、`device_sn`、`irms_sn` 路由。

### cid 生命周期入口

- `AlarmCidIndexLifecycleJob.cleanupClosedRoutes`
  - cron：`alarm.sharding.cid-index.cleanup-cron`
  - 清理 CLOSED hot/stale 路由。

- `AlarmCidIndexLifecycleJob.transferHotToStale`
  - cron：`alarm.sharding.cid-index.transfer-cron`
  - 将长期未关闭的 hot ACTIVE 路由转入 stale 表。

- `AlarmCidIndexLifecycleJob.closeExpiredStaleRoutes`
  - cron：`alarm.sharding.cid-index.stale-timeout-cron`
  - 对超过 stale 保留期的报警先写业务结束状态，再关闭路由。

### 消警可靠化入口

- `RabbitMQAlarmListener.listenMessage`
  - stop MQ 入口。
  - `alarm.sharding.enabled=true` 且 `AlarmStopEventService` 存在时，stop 消息先写 `alarm_stop_event`，成功后 MQ ack。

- `AlarmStopEventWorker.processStopEvents`
  - fixed delay：`alarm.stop-worker.normalIntervalMs`
  - 批量处理 PENDING stop event，写业务 `alarm_endTime`，关闭 cid route，标记 event APPLIED。
  - `idlePauseEnabled=true` 时，连续空轮次后进入 idle pause；空闲期间只做内存判断，低频按 `idleProbeIntervalMs` 兜底查库。

- `AlarmStopEventWorker.processSideEffects`
  - fixed delay：`alarm.stop-worker.normalIntervalMs`
  - `idlePauseEnabled=false` 时保持旧行为，独立定时执行设备恢复、电解槽扩展清理等非核心副作用事件。
  - `idlePauseEnabled=true` 时由 `processStopEvents` 同一轮顺带处理副作用，避免第二个高频查库入口。

- `AlarmStopEventWorker.cleanupAppliedEvents`
  - fixed delay：`alarm.stop-worker.cleanupIntervalMs`
  - 清理 APPLIED stop event。

- `CustomSqlLogInterceptor`
  - 拦截 MyBatis `query/update/batch`。
  - `alarm.sql-log.enabled=false` 时不打印 SQL info。
  - `mode=alarm-write` 时只打印 `AlarmMapper.insertAlarm/updateAlarm/alarmStop*` 等报警主链路写 SQL；`AlarmStopEventMapper`、`AlarmStopSideEffectMapper`、`AlarmMapper.batchStopByAlarmIds` 等 stop-worker SQL 不打印。

## 必须满足的上线前提

1. `alarm.sharding.enabled=true` 前必须执行 `src/main/resources/sql/alarm-time-capacity-sharding.sql`。
2. 必须确认旧 ShardingSphere inline 规则已经关闭，否则会出现两套分片数据源同时生效。
3. 多实例部署必须配置不同的 `alarm.sharding.id.workerId`。
4. 迁移期旧数据未处理完时保持 `alarm.sharding.includeLegacyTables=true`。
5. `alarm_cid_index.uk_alarm_cid` 默认要求 `alarm_cid` 全局唯一；如果现场存在同 cid 多设备复用，必须按 SQL 注释改成 `(alarm_cid, device_sn)` 维度并同步代码评估。
6. `alarm.profile.*` 不要作为生产开关依赖；它当前没有执行入口。

## 有误配置的表现

- `alarm.sharding.enabled` 缺失或 false：新分片、cid lifecycle job、stop worker 都不会启用。
- `alarm.sharding.maxRowsPerSlice <= 0`：应用启动创建 `AlarmIdCodec` 时失败。
- `alarm.sharding.maxRowsPerSlice > 8388608`：应用启动创建 `AlarmIdCodec` 时失败。
- `alarm.sharding.id.workerId` 不在 `0..255`：应用启动或生成 ID 时失败。
- cron 写错：对应 `@Scheduled` 任务无法按预期启动。
- `alarm.stop-worker.highIntervalMs` 配置错误：当前不会直接影响调度，因为代码未使用它作为 `@Scheduled` 间隔。
- `alarm.stop-worker.idleProbeIntervalMs` 配置过大：漏唤醒或多实例场景下 PENDING stop event 的兜底恢复会变慢。
- `alarm.sql-log.enabled=false`：所有 SQL info 都不打印，排查 SQL 时需要临时改 true。
- `alarm.sql-log.mode=all`：恢复旧的全量 SQL 打印，高流量生产环境会明显放大日志 IO。
- `alarm.profile.*` 写错：当前无影响，因为没有代码读取。
