# 报警分片数据链路说明

## 总览

`hpis-alarm` 当前链路分为两条主线：

- start 链路：MQ 开始报警 -> 分配时间容量分片 -> 写业务表 -> 写 hot cid route。
- stop 链路：MQ 停止报警 -> 写 stop event -> 唤醒 worker -> 批量关闭业务表和 route -> 生成 side effect。

核心目标是让开始报警高吞吐写入，停止报警不因线程池、远程调用或副作用变慢而丢失。

## 开始报警链路

1. 默认情况下，`RabbitMQAlarmListener` 收到 `operCode=259` 的 MQ 消息，解析 `rawData`，得到外部 `alarmId`，当前作为 `alarm_cid` 使用。
2. 如果 `alarm.batch.insertConsumerBatchEnabled=true`，旧 `RabbitMQAlarmListener` 会被条件关闭，`RabbitMQAlarmBatchListener` 接管 `alarm_queue`：
   - Spring AMQP 容器先按 `insertConsumerBatchSize/ReceiveTimeoutMs/Concurrency/Prefetch` 收集一批 `Message`。
   - listener 先解析整批消息，坏消息立即 `basicReject(false)`，不进入业务处理。
   - batch 中 start 消息先进入 `AlarmInsertBatchConsumeService` 批量持久化；start 逐条 ack/nack 后，再按原相对顺序处理 stop/status。
   - 这个顺序是为了让同批 start 先落 hot route，降低同批 stop 先执行导致 `ROUTE_MISSING` 误 FAILED 的窗口。
   - start 批量处理返回每条结果后，listener 当前线程逐条 `basicAck/basicNack`，不使用 `multiple=true`。
3. 如果 `alarm.batch.insertConsumerBatchEnabled=false`，MQ start 继续同步调用 `AlarmServiceImpl.insertAlarm`，保持旧单条 listener 语义。
4. consumer batch 链路会对每条 start 逐条解析设备缓存、断线 Redis 去重、组装 `AlarmInsertContext`；设备缓存缺失只让当前 item `DROP`，不会拖垮整批。
5. start 批量持久化规则：
   - 全批进入一个事务，跨多个 suffix 任一失败则整批回滚。
   - 批量成功后逐条返回 `SUCCESS`，并保持原有 afterCommit push。
   - 批量失败后逐条 `persistPreparedAlarmSingle(context)` 兜底，复用已准备 context，不重新执行 Redis 去重和组装副作用。
6. `AlarmServiceImpl.insertAlarm` 或 prepared 持久化链路解析 `time`，作为 `alarm_beginTime`；缺失时使用当前时间兜底。
7. `AlarmCidIndexService.allocate` 调用 `AlarmMonthlySliceTableManager` 获取本次分配结果：
   - `monthKey`：例如 `202511`。
   - `sliceNo`：例如 `0`。
   - `tableSuffix`：例如 `202511_00`。
   - `rowNo`：来自 `alarm_shard_slice` Segment 预占的行号。
8. `AlarmIdCodec` 生成可解析的内部 `alarm_id`。
9. `AlarmShardContext` 写入本次 `tableSuffix`，保证绑定表落同一组物理表。
10. 依次写入：
   - `alarm_handle_yyyyMM_nn`
   - `alarm_electrolytic_cell_yyyyMM_nn`，仅电解槽场景需要。
   - `alarm_yyyyMM_nn`
11. `alarm_cid_index` 写入 `ACTIVE` 路由：
   - `alarm_cid`
   - `alarm_id`
   - `table_suffix`
   - `device_sn`
   - `irms_sn`
   - `alarm_type`
13. 写入完成后检查同 cid 是否已有 `PENDING alarm_stop_event`，用于处理 stop 先到、start 后到的乱序。
14. push payload 仍在事务 afterCommit 后逐条发送，字段和旧消费者兼容性不变。

## 停止报警链路

1. `RabbitMQAlarmListener` 收到 `operCode=260` 的 MQ 消息。
2. 不直接同步更新业务表，而是快速 upsert 到 `alarm_stop_event`：
   - 同一 `alarm_cid` 重复 stop 只保留最新 `stop_time`。
   - 落库成功后 ack MQ。
3. `AlarmStopEventWorker` 通过定时入口处理 `PENDING` stop event；空闲时只做内存判断，收到新 stop 后被唤醒，漏唤醒时按 `idleProbeIntervalMs` 兜底扫描。
4. `AlarmStopEventService.processPendingBatch` 执行：
   - 读取 pending 数量，决定 normal/high batch size。
   - 拉取一批 pending event。
   - 按 `alarm.batch.inLimit` 拆分 `alarm_cid in (...)`，批量查询 hot/stale active route；批量 route 查询异常时才降级单条 route 查询。
   - active route 命中后按 `table_suffix` 分组；route 已 `CLOSED` 的重复 stop 直接标记 `APPLIED`；hot/stale 都找不到时先记录 `ROUTE_MISSING` 并保留 `PENDING` 重试，达到 `maxRetry` 后才标记 `FAILED`。
   - 当前没有 `PENDING` 时，会低频扫描 `FAILED/ROUTE_MISSING`；如果 start route 后到并变为 `ACTIVE`，恢复执行核心关闭，避免留下 `ACTIVE route + alarm_endTime null`。
   - 每个 suffix 分组进入核心批量更新。
5. 每个 suffix 进入 `applyRouteGroup`：
   - 设置 `AlarmShardContext`。
   - 批量更新业务表 `alarm_endTime`。
   - 批量查询已关闭的报警记录，用于生成 side effect。
   - 批量关闭 hot/stale cid route。
   - 批量标记 stop event 为 `APPLIED`。
   - 生成 side effect event。
6. `AlarmStopSideEffectService` 独立执行 side effect，不阻塞核心消警；默认由核心 worker 同一轮顺带消费，避免空闲期第二个高频查库入口。

## 查询链路

- 按新内部 `alarm_id` 查询：通过 `AlarmIdCodec` 解析日期和 slice，直达目标物理表。
- 按时间范围查询：根据 `alarm_beginTime` 范围命中对应月份和 slice。
- 按 cid 停止：只查 `alarm_cid_index` 和 `alarm_cid_stale_index` 的 active 记录。
- 长期历史：不再保证 cid 快速定位，应走内部 `alarm_id` 或时间范围。

## 清理链路

- 正常关闭后：
  - 业务分片表保留历史。
  - hot/stale route 标记 `CLOSED`，设置 `delete_after`。
  - stop event 标记 `APPLIED`，设置 `delete_after`。
- 低流量窗口：
  - 清理到期的 `CLOSED` route。
  - 清理到期的 `APPLIED alarm_stop_event`。
- 长时间未关闭：
  - hot route 转 stale route。
  - stale 到期前必须先执行业务超时关闭，再删除 route。

## 异常与幂等

- stop 先到、start 后到：stop event 保持 `PENDING`，start 写入 route 后立即补偿关闭。
- 重复 stop：如果 route 已经 `CLOSED`，stop event 可直接批量标记 `APPLIED`。
- side effect 失败：只影响副作用事件状态，不回滚核心 `alarm_endTime`。
- route 缺失：批量链路不扫描历史分片，hot/stale 都找不到时先保持 `PENDING` 并记录 `last_error=ROUTE_MISSING`；达到 `maxRetry` 后才转 `FAILED`，后续 route 如果出现会被恢复扫描重新关闭。
- worker 漏唤醒：`idleProbeIntervalMs` 兜底查库会重新发现 `PENDING`，不会只依赖内存状态。
- MQ start consumer batch：消息在 batch listener 返回最终结果前仍处于未 ack；DB commit 成功但 ack 前宕机时依赖 RabbitMQ 重投和 `alarmCid` 幂等兜底，灰度前必须确认不会重复插入主报警。
- consumer batch 同批混合消息：start 会先批量持久化，再处理同批 stop/status，降低 route 尚未写入时 stop 被误判 `ROUTE_MISSING` 的窗口。
- 断线 Redis 去重：batch 批量失败后拆单兜底复用 `AlarmInsertContext`，不会重新 `tryAcquire`；只有最终入库失败的 prepared context 才释放本次 acquired key。
- Rabbit 并发顺序：旧 listener `concurrency="10-36"` 或 batch listener `insertConsumerBatchConcurrency` 下，同一 `alarmCid` 的 start 消息都不保证严格顺序，第一阶段沿用当前最终一致语义，不新增同 cid 串行锁。

## 2026-05-24 alarm_id 生成边界补充

- `AlarmServiceImpl.insertAlarmSingle` 和 `AlarmServiceImpl.buildAlarmInsertContext` 会在扩展表组装前生成内部 `alarm_id`，外部 `rawData.alarmId` 仍只作为 `alarm_cid`。
- `alarm_handle`、`alarm_electrolytic_cell`、`alarm_electrolytic_cell_ectype`、`alarm_cid_index` 的 `alarm_id` 必须以 `alarm.alarmId` 为唯一来源。
- 批量链路会把生成好的 `shardSuffix` 保存进 `AlarmInsertContext`；批量事务和批量失败拆单兜底都复用同一个 `alarm_id/shardSuffix`，不能重新分配。
- 电解槽扩展对象入库前会调用 `bindElectrolyticCellToAlarm` 做最后绑定；该方法只同步主表身份字段，不重新读取 Redis，也不改变 MQ 消息体和 push payload。
