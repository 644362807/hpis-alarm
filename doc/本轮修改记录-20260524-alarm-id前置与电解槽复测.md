# 本轮修改记录-20260524-alarm-id前置与电解槽复测

## 修改背景

本轮定位到电解槽报警在 `alarm_electrolytic_cell` 入库时可能拿不到内部 `alarm_id`。根因是 `AlarmServiceImpl.prepareAlarmShard` 才会生成主表内部 `alarm_id`，但电解槽扩展对象原来在该方法之前组装，导致扩展表以空 `alarm_id` 进入 ShardingSphere 分片计算。

## 修改范围

- `AlarmServiceImpl.insertAlarmSingle`
  - 在 `jsonTransformJava` 之后、处理表和电解槽扩展表组装之前调用 `prepareAlarmShard`。
  - 失败且未进入完整入库时主动清理 `AlarmShardContext`。
- `AlarmServiceImpl.buildAlarmInsertContext`
  - 批量 prepare 阶段提前生成内部 `alarm_id` 和 `shardSuffix`。
  - 将 `shardSuffix` 保存到 `AlarmInsertContext`，后续批量事务和拆单兜底复用。
- `AlarmServiceImpl.persistAlarmInsertContexts`
  - 优先复用 context 中已经分配好的 `shardSuffix/alarm_id`，只有缺失时才兜底重新分配。
- `AlarmServiceImpl.persistAlarmInsertGroup`
  - 批量入库电解槽扩展表前调用统一绑定方法。
- `AlarmServiceImpl.persistPreparedAlarmSingleInTransaction`
  - 批量失败拆单兜底时复用同一个 `alarm_id` 绑定电解槽扩展表。
- `AlarmServiceImpl.insertSql`
  - 旧单条链路优先复用前置生成的 `AlarmShardContext`，避免重复分配内部 ID。
- `AlarmServiceImpl.bindElectrolyticCellToAlarm`
  - 新增内部方法，入库前以 `alarm` 主表对象为准同步 `alarm_id/alarm_beginTime/createTime/irmsSn`。

## 外部入口是否变化

不变化。Controller、MQ queue、MQ 消息体、`insertAlarm(JSONObject)`、`insertAlarms(List<JSONObject>)` 的方法签名都不变。

## 数据库表结构是否变化

不变化。本轮只修正多表写入时 `alarm_id` 的生成时机和绑定来源，不新增字段、索引或表。

## MQ 消息体和 push payload 是否兼容

兼容。外部 `rawData.alarmId` 仍作为 `alarm_cid` 使用；push 仍 afterCommit 逐条发送旧 JSON 结构。

## 幂等风险

- 断线 Redis 去重逻辑不变。
- 批量失败后拆单兜底仍复用 `AlarmInsertContext`，不重新执行 Redis `tryAcquire`。
- 风险边界：`prepareAlarmShard` 前置后，如果后续设备扩展信息组装失败，会比过去更早消耗一个分片 rowNo，但不会写入业务表；这是低风险空洞，不影响主键唯一性。

## 数据一致性风险

- 修复目标是保证 `alarm`、`alarm_handle`、`alarm_electrolytic_cell`、`alarm_electrolytic_cell_ectype`、`alarm_cid_index` 的 `alarm_id` 一致。
- 本轮没有改变写表顺序，仍保留原先“扩展表/处理表先写，主表后写”的历史顺序；若后续要进一步降低半成品风险，应单独评审是否调整写表顺序。

## 可观测性变化

本轮不新增配置开关；通过已有 `logAlarmInsertStage` 和异常日志观察 `BUILD_ALARM/PERSIST/INSERT_ALARM` 阶段。电解槽 Redis 缓存缺失仍会打印原有错误日志。

## 配置开关和回滚方式

无新增配置。回滚方式是回退本轮 `AlarmServiceImpl` 中前置 `prepareAlarmShard`、复用 `context.shardSuffix`、`bindElectrolyticCellToAlarm` 相关修改。

## 已执行验证

待执行：

- `mvn -pl hpis-alarm -DskipTests compile`
- 使用完整电解槽报警报文进行 MQ start 入库复测。
- 校验 `alarm` 与 `alarm_electrolytic_cell` 的 `alarm_id` 一致。

## 未覆盖风险和下一步建议

- 当前 `insetElectrolyticCell` 在 Redis 电解槽序列缓存缺失时只记录错误并返回空对象，修复 `alarm_id` 后仍可能写入字段不完整的电解槽扩展记录。建议下一轮单独评审：缓存缺失时是 DROP、FAIL requeue，还是落主表但不落扩展表。
- 电解槽扩展表仍依赖 `irmsSn + seq` Redis key，复测必须使用包含 `irmsSn`、`seq`、`type`、`subdivideIndex`、`kua`、`grooveIndex`、`maxTemp` 的完整报文。
