# 报警停止可靠化与批量关闭设计说明

## 设计目标

现场高流量场景中，1 万 start + 1 万 stop 可能让消警链路长时间堆积，线程池队列满后出现 stop 丢失或 `alarm_endTime` 长时间缺失。

第一阶段固定采用：

- MQ 手动 ack。
- stop event 可靠落库。
- 后台 worker 批量关闭。
- side effect 独立执行。
- 空闲时暂停 worker 查库，新 stop 入库后唤醒。

## stop event 表定位

`alarm_stop_event` 是短生命周期可靠缓冲表，不是历史表。

状态：

- `PENDING`：stop 已收到，但核心消警尚未完成。
- `APPLIED`：业务表和 cid route 已关闭。
- `FAILED`：超过最大重试次数，需要人工排查。

同一 `alarm_cid` 重复 stop 时，通过 upsert 保留最新 `stop_time`，避免重复堆积。

## MQ ack 策略

- start 消息：业务插入成功后 ack；异常时 nack/requeue。
- stop 消息：写入 `alarm_stop_event` 成功后 ack；异常时 nack/requeue。

这样 stop 不依赖业务关闭是否立即完成，只要 stop event 落库，最终 worker 会继续处理。落库成功后会唤醒空闲 worker；如果发生漏唤醒、重启或多实例路由不均，低频兜底扫描仍会重新发现 `PENDING`。

## 批量关闭流程

`AlarmStopEventService.processPendingBatch` 负责核心批处理：

1. 读取 pending 数量，决定普通批量或高流量批量。
2. 拉取一批 `PENDING` event。
3. 按 `alarm.batch.inLimit` 把 `alarm_cid` 拆成多段 `IN` 查询，批量读取 hot/stale active route；只有批量 route 查询异常时才降级单条 route 查询。
4. active route 命中后按 `table_suffix` 分组，避免一个批次跨多个物理表更新；route 已 `CLOSED` 的重复 stop 直接标记 `APPLIED`；hot/stale 都查不到时批量标记 `FAILED`，`last_error=ROUTE_MISSING`。
5. 每个 suffix 设置 `AlarmShardContext`。
6. 批量更新 `alarm_endTime` 和 `alarm_status`。
7. 批量查询业务报警，用于生成 side effect。
8. 按 hot/stale 拆分后批量关闭 cid route。
9. 批量标记 stop event 为 `APPLIED`。
10. 生成 side effect event。

## 空闲暂停与唤醒

`AlarmStopEventWorker` 仍保留 `@Scheduled` 入口，但默认启用 `alarm.stop-worker.idlePauseEnabled=true`：

- worker 启动默认 active，保证应用重启后能先扫描已有 `PENDING`。
- 连续 `idleConfirmCount` 轮核心消警和 side effect 都无处理量后进入 idle pause。
- idle pause 后 scheduled 方法只做内存判断，不再每秒查询 `alarm_stop_event` 或 `alarm_stop_side_effect_event`。
- `AlarmStopEventService.recordStop` 在 stop event 落库成功后唤醒 worker。
- `applyPendingStopForNewAlarm` 补偿成功后也会唤醒 worker，避免新生成的 side effect 等待兜底扫描。
- `idleProbeIntervalMs` 控制低频兜底查库，默认 60000ms，防止漏唤醒或多实例场景下 `PENDING` 长期卡住。

## 本轮批量化点

| 批量点 | 原方式 | 新方式 |
| --- | --- | --- |
| route 查询 | 每条 stop event 查询 hot/stale route | 按 `alarm.batch.inLimit` 做 `alarm_cid in (...)` 批量查询 |
| route 关闭 | 每条 `closeHotByAlarmId/closeStaleByAlarmId` | `closeHotBatch/closeStaleBatch` |
| stop event 完成 | 每条 `markApplied` | `markAppliedBatch` |
| route 缺失失败 | 保留 `PENDING` 等待 start 补偿 | 批量 `markFailedBatch`，`last_error=ROUTE_MISSING` |
| side effect 完成 | 每条 `markDone` | `markDoneBatch` |

route 批量关闭使用：

```sql
alarm_endTime = case alarm_id
  when ? then ?
  when ? then ?
end
```

这样同一批内每条报警仍写入自己的 `stop_time`，不会被统一时间覆盖。

## side effect 策略

核心消警只保证：

- 业务表 `alarm_endTime` 写入。
- cid route 关闭。
- stop event 标记 `APPLIED`。

设备恢复、电解槽扩展清理、推送等都作为 side effect：

- side effect 失败不会回滚核心消警。
- 高流量时可以延迟执行。
- 成功 event id 收集后批量 `markDoneBatch`。

`EC_ECTYPE_DELETE` 只允许 `sceneType=2` 的电解槽报警生成。为了保证判断正确，批量关闭后补查业务报警时必须包含 `scene_type` 字段。

## 幂等与乱序

- stop 先到、start 后到：stop event 保持 `PENDING`，start 成功后调用补偿路径立即关闭。
- 重复 stop：如果 route 已经 `CLOSED`，event 可直接标记 `APPLIED`。
- 找不到 active route：批量链路会继续查 hot/stale 全状态 route；如果已 `CLOSED`，按重复 stop 标记 `APPLIED`；如果 hot/stale 都查不到，直接转 `FAILED`，`last_error=ROUTE_MISSING`。这是本轮明确接受的业务变化。
- stop 先到、start 后到补偿仍保留在 start 写入 route 后的 `applyPendingStopsForNewAlarms`；但如果 worker 已提前把该 stop 判定为 `ROUTE_MISSING`，不会再被 start 后补偿捞起。
- side effect 重复生成：通过 `alarm_id + effect_type` 唯一约束避免重复执行。

## 高低流量策略

低流量：

- 小批量处理。
- side effect 正常执行。
- 清理到期 `APPLIED` stop event 和 closed route。
- 连续空轮次后进入 idle pause，减少无任务时的 DB 查询和 SQL 日志。

高流量：

- 提高 stop event 批量大小。
- cleanup 暂停或降频。
- side effect 不阻塞核心消警。

当前压测显示核心批量化有效，但剩余耗时主要在 MQ 消费节奏、start 插入链路、stop event upsert 和电解槽 side effect。

## SQL 日志边界

stop-worker 的普通完成日志受 `alarm.stop-worker.logEnabled` 控制；SQL 日志不归它控制。SQL info 由 `alarm.sql-log.*` 控制：

- `enabled=false`：不打印 SQL info。
- `mode=alarm-write`：只打印报警主链路写 SQL 和非 worker 慢 SQL。
- `mode=all`：恢复旧的全量 SQL 打印，不建议生产长期开启。
- stop-worker 相关 mapper SQL 默认不打印，只保留异常日志兜底。
