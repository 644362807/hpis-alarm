# 报警时间容量分片设计说明

## 设计目标

原规则按 `alarm_id % 5` 固定分到 `alarm_0..4`、`alarm_handle_0..4`、`alarm_electrolytic_cell_0..4`。该方式无法按时间治理历史数据，也无法在单月数据暴涨时继续切分。

当前方案改为时间 + 容量分片：

- 按 `alarm_beginTime` 先定位月份。
- 单月内按容量切 slice。
- 物理表后缀为 `yyyyMM_nn`，例如 `alarm_202511_00`。
- 三张绑定表使用同一 `table_suffix`。

## 物理表规则

| 逻辑表 | 物理表格式 | 示例 |
| --- | --- | --- |
| `alarm` | `alarm_yyyyMM_nn` | `alarm_202511_00` |
| `alarm_handle` | `alarm_handle_yyyyMM_nn` | `alarm_handle_202511_00` |
| `alarm_electrolytic_cell` | `alarm_electrolytic_cell_yyyyMM_nn` | `alarm_electrolytic_cell_202511_00` |

`nn` 是月内 slice 序号，从 `00` 开始。单月当前 slice 达到 `alarm.sharding.maxRowsPerSlice` 后，创建下一组物理表。

## 核心元数据

### `alarm_shard_slice`

用于记录每个月每个 slice 的容量分配状态。

关键字段：

- `month_key`：月份，例如 `202511`。
- `slice_no`：月内 slice 序号。
- `table_suffix`：物理表后缀，例如 `202511_00`。
- `current_rows`：已预占行数，不等于真实行数。
- `max_rows`：当前 slice 最大容量。

### `alarm_cid_index`

hot cid 路由表，只服务正在报警和刚关闭待清理的记录。

关键字段：

- `alarm_cid`：外部报警 ID。
- `alarm_id`：内部可解析 ID。
- `table_suffix`：业务分片后缀。
- `route_status`：`ACTIVE` 或 `CLOSED`。
- `delete_after`：关闭后允许物理清理的时间。

### `alarm_cid_stale_index`

stale cid 路由表，用于长时间未关闭报警。它不是永久历史索引，过期前必须先对业务报警做超时关闭，再删除路由。

## ShardingSphere 配置口径

- 保留 ShardingSphere 4.1.1，不升级 5.x。
- 复用 `hpis-quad` 类似的 Java API 动态建表思路。
- Nacos 中旧的 `alarm_id % 5` inline 规则需要关闭，避免两套分片数据源同时生效。
- 使用 `spring.shardingsphere.enabled=false` 禁用 starter 自动配置，由业务 Java 配置创建分片数据源。

建议 Nacos 核心配置：

```yaml
spring:
  shardingsphere:
    enabled: false
    datasource:
      names: ds
      ds:
        type: com.zaxxer.hikari.HikariDataSource
        driver-class-name: com.mysql.cj.jdbc.Driver
        url: jdbc:mysql://127.0.0.1:3306/hpis_alarm?useUnicode=true&characterEncoding=utf8&serverTimezone=Asia/Shanghai
        username: root
        password: 123456

alarm:
  sharding:
    enabled: true
    maxRowsPerSlice: 5000000
    allocationSegmentSize: 1000
    id:
      workerId: 0
```

## 路由优先级

1. 写入链路优先使用 `AlarmShardContext`，确保绑定表同 suffix。
2. 按 `alarm_id` 查询时优先解析内部 ID，直达目标表。
3. 按 cid 停止时查 hot/stale cid route。
4. 按时间范围查询时根据月份和 slice 命中业务分片。
5. 无有效分片键时可能广播，应在接口层尽量带时间条件或 ID 条件。

## 当前边界

- 不维护永久 cid 历史索引。
- 删除 cid route 后，不能仅凭 cid 快速查长期历史。
- `alarm_cid` 默认基本唯一；若现场存在重复，应将唯一键调整为 `(alarm_cid, device_sn)`。
- 迁移期间旧数据可通过 legacy fallback 受控兼容，新数据必须走新路由。
