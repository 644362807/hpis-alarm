# 报警 MQ-MySQL 批量消费压测阶段报告

测试日期：2026-05-24

## 结论

本轮使用本机 Nacos 配置启动 `hpis-alarm`，并在真实 RabbitMQ + MySQL 环境下执行 2000 到 100000 档压测。普通报警“先 start 后 stop”场景已通过 `2000 / 10000 / 50000 / 100000` 四档；普通报警“start/stop 交替上报”在 `50000` 档出现一致性失败，因此后续 `100000`、电解槽、混合场景和分片上限测试已暂停。

阻断问题不是 MQ 堆积未消费，而是 stop worker 在高并发交替上报下过早把 3 条 stop event 标记为 `FAILED/ROUTE_MISSING`。后续 SQL 复核显示对应 `alarm_cid_index` 路由已经存在且仍为 `ACTIVE`，主表报警也存在但 `alarm_endTime` 为空，说明该场景存在 start/stop 乱序或路由可见性窗口下的误失败风险。

## 环境与配置

- 服务启动：使用 Nacos 中现有配置。
- 服务端口：`8806`。
- JVM：Java 8，额外增加 `-Dfile.encoding=UTF-8`，用于避免 Java 8 默认 GBK 读取 Nacos YAML 失败。
- RabbitMQ：`127.0.0.1:5672`，队列 `alarm_queue`。
- MySQL：`127.0.0.1:3306`，库 `hpis_alarm`。
- consumer batch：已启用。
- 本轮关键参数：`batchSize=100`，`receiveTimeoutMs=50`，`prefetch=100`，`concurrency=2-4`。
- 本地压测 stub：已启用，用于避开 push、WebSocket、跨服务调用对 MQ -> DB 主链路的影响。

## 失败详情

失败 run：`LT-GALT-50K-20260524182348-2`

最终状态：

- RabbitMQ：`alarm_queue.ready=0`。
- 主表：`alarmRows=50000`。
- 已关闭：`closedRows=49997`。
- stop event：`APPLIED=49997`，`FAILED=3`。
- hot route：`CLOSED=49997`，`ACTIVE=3`。

SQL 复核失败事件：

| alarm_cid | event_status | last_error | retry_count | alarm_id | alarm_endTime | route_status | table_suffix |
| --- | --- | --- | ---: | ---: | --- | --- | --- |
| `{P182348GALT50K2-00mvb}` | FAILED | ROUTE_MISSING | 1 | 657244070628029360 | NULL | ACTIVE | `202605_00` |
| `{P182348GALT50K2-00a0v}` | FAILED | ROUTE_MISSING | 1 | 657244070619647309 | NULL | ACTIVE | `202605_00` |
| `{P182348GALT50K2-00qi8}` | FAILED | ROUTE_MISSING | 1 | 657244070619642803 | NULL | ACTIVE | `202605_00` |

判断：stop event 被标记 `FAILED/ROUTE_MISSING` 后不会再自动恢复；但对应路由后来存在且保持 `ACTIVE`，导致报警主表形成 3 条未闭合记录。这不符合验收标准中的 `alarm_stop_event.FAILED = 0` 和 `closedRows >= expectedStopCount`。

## 暂停项

以下测试因上述一致性失败暂停：

- `GENERAL_ALTERNATE + 100000`
- `ELECTROLYTIC_START_THEN_STOP`
- `ELECTROLYTIC_ALTERNATE`
- `MIXED_START_THEN_STOP`
- `MIXED_ALTERNATE`
- 分片上限测试：`MIXED_ALTERNATE + 100000`，`alarm.sharding.maxRowsPerSlice=10000`

## 风险判断

- ID 一致性：本轮普通报警场景未发现 `alarm_id` 为空或主扩展表 ID 不一致问题；电解槽场景尚未进入本轮高压矩阵。
- 缺参毒消息：本轮未重新执行电解槽缺参毒消息验证。
- MQ 重投：失败 run 终态 `queueReady=0`，没有持续重投现象；问题落在 DB stop event 状态机。
- MySQL 半成品数据：`GENERAL_ALTERNATE 50000` 出现 3 条主表已插入但未闭合的业务半成品，需要修复后重跑。

## 建议修复方向

后续如需改代码，按约定先切测试分支。建议把 `ROUTE_MISSING` 从立即失败调整为可重试状态：当 stop worker 当前批次查不到 route 时，先保持 `PENDING` 或带退避重试，不立即 `FAILED`；只有超过重试次数、超过时间窗口，或能确认该 cid 永远不会产生 route 时再失败。同时对已经 `FAILED/ROUTE_MISSING` 但后来 route 变为 `ACTIVE` 的记录增加恢复路径，避免高并发交替上报留下未闭合报警。

修复后建议从 `GENERAL_ALTERNATE 50000` 重新开始，验证通过后再继续 `100000`、电解槽、混合场景和 `maxRowsPerSlice=10000` 分片上限测试。

## 2026-05-24 修复进展

已在测试分支 `test/alarm-batch-perf-diagnostics` 落地第一步修复：

- `ROUTE_MISSING` 默认改为 `PENDING + retry`，达到 `alarm.stop-worker.maxRetry` 后才转 `FAILED`。
- 增加 `FAILED/ROUTE_MISSING` 恢复扫描；如果 route 后到并变为 `ACTIVE`，会重新走核心关闭链路。
- verifier 报告新增 `verifierStartupGap`、首次观测、队列清空、DB 闭合、可信闭环耗时，避免 2000 档因启动空窗被误判。
- 新增 orchestrator 压测入口，后续可先启动 verifier 再发 MQ。
- 已通过目标单测和父工程边界编译；仍需重启服务后从 `GENERAL_ALTERNATE 50000` 重新压测确认。
