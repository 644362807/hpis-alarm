# 报警 SQL 循环与批处理风险治理计划

## 背景

报警模块同时承载 MQ start 入库、stop 最终一致关闭、分片路由生命周期和配置维护。积压放大后，循环内逐条 SQL 会形成 N+1 往返；集合未经切块直接进入 `IN`、`CASE WHEN` 或多 values `INSERT`，又会形成超长 SQL、长事务和锁等待。

本轮建立统一规则：单条 SQL、单次 JDBC batch 和单个事务内的业务写批次默认不得超过 `500` 条。涉及物理分表时，必须先按 `table_suffix` 分组，再在分片内切块。

## 已治理风险

| 优先级 | 风险点 | 原问题 | 本轮处理 | 状态 |
| --- | --- | --- | --- | --- |
| P0 | stop 批量异常降级 | route 批量查询失败后可能逐条查询整批 hot/stale route | 增加 `fallbackSingleMaxItems` 保险丝，默认最多 `100` 条；超限整体失败并保留重试语义 | 已完成 |
| P0 | start 批量失败降级 | 批量插入失败后可能逐条事务重放整批 | 同一保险丝限制最多 `100` 条；超限返回 `FAIL`，由 MQ nack/requeue | 已完成 |
| P0 | `alarm.batch.inLimit` | 只有小于等于 0 兜底，没有最大值保护 | 新增 `AlarmBatchChunker.safeBatchSize()`，统一钳制为 `1-500`，非法值按 `500` | 已完成 |
| P1 | cid route 批量关闭 | `closeRoutes()` 循环逐条更新 hot/stale route | 按 hot/stale 分组，每组按 `500` 切块执行批量关闭 | 已完成 |
| P1 | hot 转 stale | 每条 route 单独 upsert stale，再 delete hot | 批量 upsert stale 后批量 delete hot；事务内保持原幂等顺序 | 已完成 |
| P1 | stale 超时关闭 | 循环逐条更新物理 alarm 和 route | 先按 `table_suffix` 分组，再按 `500` 切块关闭业务表和 route | 已完成 |
| P1 | start 同 suffix 入库 | 依赖 MQ batch 较小，方法自身没有硬边界 | 主表、处理表、扩展表和 hot route 均在 suffix 内按 `500` 切块 | 已完成 |
| P1 | stop 同 suffix 关闭 | 主路径有外层 chunk，但旧路径可绕过 | `applyRouteGroup()` 方法自身再次按 `500` 切块，所有入口统一收敛 | 已完成 |
| P1 | 确认报警定时任务 | Redis Map 可能很大，更新 SQL 的 `IN` 未切块 | 过期 alarm id 按 `500` 切块更新 | 已完成 |
| P1 | 报警配置设备关联 | 租户设备数量较大时批量 insert 未切块 | 设备关联按 `500` 切块 insert | 已完成 |
| P1 | side effect 完成回写 | `normalBatchSize` 配置误放大时 `markDoneBatch` 可能超过 SQL 边界 | select limit 和 `markDoneBatch` 均按 `500` 硬钳制 | 已完成 |
| P2 | 配置时间段新增 | 循环逐条 insert | 改为多 values 批量 insert，并按 `500` 切块 | 已完成 |

## 保留例外与后续台账

| 优先级 | 风险点 | 当前状态 | 保留原因 | 后续方案 |
| --- | --- | --- | --- | --- |
| P2 | stop 小批异常兜底 | route 查询失败后允许最多 `100` 条逐条查询 | 只用于短时异常定位和兼容恢复；已有限额和日志 | 第二阶段 PROCESSING claim 上线后继续观察触发频率，长期目标是消除常态触发 |
| P2 | start 小批异常兜底 | 批量插入失败后允许最多 `100` 条逐条事务重放 | 用于隔离单条坏数据；复用 prepared context，避免 Redis 去重重复执行 | 灰度监控 `FALLBACK_SINGLE` 次数和耗时，持续触发时修根因 |
| P2 | side effect 小批异常兜底 | 批量事件生成失败后允许最多 `100` 条逐条创建 | side effect 不参与核心消警成功判定 | 第二阶段核心闭环通过后，为 side effect event 增加 PROCESSING claim |
| P2 | 人工重发记录 | 远程调用仍逐条执行 | 下游接口不是批量接口，且需要单条隔离失败 | 批量预查本地数据，远程调用保留低并发，状态更新按 `500` 切块 |
| P2 | Controller 批量删除 | 部分 Mapper 仍直接接收数组拼 `IN` | 属于管理端低频入口，不阻塞本轮 MQ 核心链路 | 增加请求上限和统一 chunk，单独回归接口权限与返回语义 |

## 第二阶段继续处理

以下问题不是简单切块可以解决，进入 stop 闭环专项：

1. `countPending` 在热路径反复全量 count，75 万积压时会持续消耗索引扫描成本。
2. `selectPendingBatch` 没有认领状态，多线程直接并行会重复取同一批事件。
3. stop worker 目前由单个 scheduled 入口串行执行多个批次，`maxParallelism` 实际不是并行线程数。
4. route missing 需要增加 `available_time` 延迟，避免坏消息热循环。
5. `mark APPLIED` 需要增加 `PROCESSING + lock_token` 条件，避免并发线程误更新其它线程认领的数据。
6. verifier 高频完整 count 会干扰性能结论，需要调整为轻量热采样、终态完整校验。

## 统一实现

- 新增 `AlarmBatchChunker`：统一处理硬上限、非法值兜底和列表切块。
- 新增 `alarm.batch.fallbackSingleMaxItems`：默认 `100`，非法值按 `100`，最大值钳制为 `500`。
- 统一日志：
  - `FALLBACK_SINGLE_REJECTED`
  - `SIDE_EFFECT_FALLBACK_SINGLE_REJECTED`
- 规则已同步写入根目录 `agent.md`，后续修改必须专项检查循环 SQL 和无 chunk 超大 SQL。
