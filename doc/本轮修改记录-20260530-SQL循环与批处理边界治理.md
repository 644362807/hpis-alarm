# 本轮修改记录：SQL 循环与批处理边界治理

## 修改背景

stop 积压达到几十万条时，循环逐条 SQL 和无硬边界的批量 SQL 都会放大数据库压力。本轮先治理可直接收敛的 P0/P1 风险，为后续 PROCESSING claim 并行 worker 建立稳定的小事务基础。

## 修改范围

- 新增 `AlarmBatchChunker`，统一将单批数量限制在 `1-500`。
- `AlarmBatchProperties` 新增 `fallbackSingleMaxItems=100`，并为 `inLimit` 增加最大值保护。
- start 批量失败、stop route 查询失败、side effect 事件批量创建失败的逐条降级均增加保险丝。
- start 同 suffix 入库、stop 同 suffix 关闭、cid route 关闭、hot 转 stale、stale 超时关闭、side effect 完成回写、确认报警更新、报警配置设备关联和时间段新增均增加 chunk。
- 根目录 `agent.md` 增加 SQL 批处理设计规则。

## 兼容性

- Controller、MQ queue、MQ 消息体、Redis key、push payload 和公开 service 方法签名不变。
- 数据库表结构不变。
- MQ ack 语义不变：start 超限降级拒绝后返回 `FAIL`，仍由 listener nack/requeue。

## 一致性与幂等

- hot 转 stale 仍保持“先 upsert stale，再 delete hot”的事务顺序。
- 涉及物理表的操作仍先按 `table_suffix` 分组，再在分片内切块。
- 批量失败拆单仍复用已经准备好的 start context，不重新执行 Redis 去重。
- side effect 失败不反向回滚核心 `alarm_endTime` 和 route CLOSED。

## 可观测性

新增超限拒绝日志：

- `FALLBACK_SINGLE_REJECTED`
- `SIDE_EFFECT_FALLBACK_SINGLE_REJECTED`

## 配置与回滚

- 新配置：`alarm.batch.fallbackSingleMaxItems`，默认 `100`，最大 `500`。
- 回滚时可关闭 `alarm.batch.fallbackSingleOnBatchError`，完全停止逐条降级。
- `alarm.batch.inLimit` 即使误配置超过 `500`，运行时仍会钳制为 `500`。

## 已执行验证

Java 8 + UTF-8 focused 单测通过：`20` 个用例，`0` 失败，`0` 错误。补充 side effect 边界测试后，全量 `hpis-alarm` 单测通过：`50` 个用例，`0` 失败，`0` 错误，`2` 个历史跳过。

真实 RabbitMQ + MySQL 回归通过：

- `SQLGOV-GALT-10K-20260531000155`：`GENERAL_ALTERNATE 10000`，`True closed-loop elapsed=57859ms`，最终 `alarmRows=closedRows=APPLIED=10000`。
- `SQLGOV-MALT-10K-20260531000438`：`MIXED_ALTERNATE 10000`，`True closed-loop elapsed=70783ms`，最终 `alarmRows=closedRows=APPLIED=10000`，`ecRows=4500`，`ecMissing=0`。
- 两轮最终 `alarm_queue.ready=0`，未观察到持续 nack/requeue、alarm id 不一致、死锁或锁等待。

真实启动前发现本地 `target/classes/mapper` 存在陈旧生成资源，导致旧 jar 缺少 `selectFailedRouteMissingBatch`。源码 XML 已正确；清理明确的陈旧生成副本并重新打包后恢复。该经验已写入 `agent.md`，`target/` 不纳入提交。

## 未覆盖风险

- `countPending/selectPendingBatch` 热路径、串行 stop worker 和 route missing 热重试尚未解决。
- 管理端 Controller 批量删除、人工重发记录仍在 P2 台账中。
- 真实 RabbitMQ + MySQL 回归压测需要在第二阶段代码完成后按低档到高档执行。
